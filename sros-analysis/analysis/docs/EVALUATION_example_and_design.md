# 예시·설계 평가 및 보완 요약

## 1. 예시(expansion) 평가

### 1.1 적절한 예시인가?

**적절함.**  
- 주제가 명확(계정 가전 확장, 이항·횟수 결과).  
- Level 0→5 전 구간을 한 번에 보여줌.  
- 설계(재현성 명세, 시각화 전략, Phase lock)를 따르는 형태로 작성됨.  
- 연관만 기술하고 인과 미허용이 01_charter에 명시됨.

**한계:**  
- 수치는 “가정”이라 실제 코드/SQL 출력이 없음.  
- “왜 이 순서로 하는지”, “각 단계에서 무엇을 얻는지”가 예시 문서만으로는 초심자에게 직관적이지 않을 수 있음.

### 1.2 사전 지식 없는 사람이 읽을 때

**이해를 돕기 위해 부족한 점:**  
- **절차 맥락:** Level 0~5가 무엇을 위한 단계인지, 한 문장으로 정리된 “이번 단계에서 하는 일”이 예시 내에 없음.  
- **용어:** estimand, 오즈비(OR), rate ratio, ΔAIC, 과분산, VIF, Phase lock 등이 예시 문서에서 정의 없이 쓰임.  
- **결과 해석:** “오즈비 1.28”이 무슨 의미인지, “95% CI를 왜 쓰는지”에 대한 한두 문장 설명이 없음.  
- **납득 구조:** “증거 규칙(R1)”, “불확실성 필수(R3)”, “Phase lock(R0)”이 결과를 납득시키는 장치라는 설명이 예시 안에 없음.

**보완 방향:**  
- 예시 폴더에 **00_reader_guide.md** 추가: “무엇을 수행하는지”, “어떤 순서로 읽으면 되는지”, “핵심 용어”, “결과를 어떻게 받아들이면 되는지” 요약.  
- 설계에 **00_process_overview.md**(Level 0~5 한 페이지 요약), **glossary.md**(핵심 용어 1줄 정의) 추가해 예시가 참조하도록 함.

### 1.3 데이터 분석 절차의 합당성·결과 납득 가능성

**절차:**  
- 데이터 계약(02) → 품질·특성·특징 피처(03) → 가설(04) → 모델 설계(05) → 평가(06) → 결정(07) → 보고(08) 순서가 논리적.  
- “결과 변수 유형 → 모델 선택”(이항→Logit, 횟수→Poisson/NB)이 설계와 일치.  
- Phase lock, 증거 규칙, 불확실성(95% CI) 요구가 문서에 반영됨.

**결과 납득:**  
- 추정치에 95% CI가 붙어 있음.  
- 한계(단면, 혼동, 인과 아님)가 08과 03 §7에 명시됨.  
- “왜 이 절차를 따르면 결과를 믿을 수 있는가”를 **예시 내에서** 한두 문장으로 짚어주면 납득이 쉬워짐 → 00_reader_guide에 “납득 구조” 문단 추가.

## 3. 설계 보완 사항(반영할 내용)

- **00_process_overview.md:** 각 Level의 목표, 입력, 산출물, 한 줄 설명.  
- **templates/01_charter.md:** 분석명, 연구 질문, 인과 허용 여부, 범위 등 최소 필드.  
- **glossary.md:** 분석/설계에서 쓰는 핵심 용어 1줄 정의.  
- **예시 측:** 00_reader_guide.md — “무엇을 수행하는지”, “읽는 순서”, “용어”, “결과 납득 구조” 요약.

## 4. rules / skills / subagent 보완

- **Rules:** 설계 문서 참조만 유지. “실행 폴더 README 또는 00_reader_guide에 수행 절차 요약을 두어 초심자가 따라가기 쉽게 할 것”은 설계·예시 측 권장으로 두고, rules에는 필수 규칙으로 넣지 않음.  
- **Skills:** “실행 폴더에 독자 가이드(00_reader_guide 또는 README 수행 절차 요약)를 두면 좋음”을 REFERENCE 또는 보고 절에 추가. design/00_process_overview, design/glossary 참조.  
- **Subagent:** WORKFLOW 전에 “실행 폴더 구성” 문단 추가 — 00_reader_guide 또는 README 수행 절차 요약 권장, process_overview·glossary 참조.
