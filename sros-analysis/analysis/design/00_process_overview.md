# 00 Process Overview — Level 0~5 한 페이지 요약

**목적:** 사전 지식이 없는 독자도 “무엇을 수행하는 절차인지”를 한눈에 파악할 수 있도록 각 단계의 목표·입력·산출물을 요약한다.

---

## 전체 흐름 (고정 순서)

**데이터 계약 → 데이터 특성·품질·특징 피처 → 가설 도출 → 모델 설계 → 모델 적합·평가 → 결정·로버스트니스 → 최종 보고서.**  
단계를 건너뛰지 않는다(Phase lock).

| Level | 목표 (한 줄) | 주요 입력 | 주요 산출물 |
|-------|----------------|-----------|-------------|
| **0** | 데이터가 무엇을 담고, 어떻게 정의되는지 계약하고 품질을 점검한다. | 테이블/스키마, 시간 기준, 결과 정의 | 02_data_contract, sql/00_checks, 03 §2–§3 |
| **1** | 데이터 특성을 기술하고, 가설 후보(특징 피처)를 뽑아 가설 트리를 만든다. | 02, 03 §2–3, 집계·시각화 | 03 §4–§5(특징 피처), 04_hypothesis_tree |
| **2** | 가설에 맞는 모델(회귀/로지스틱/Poisson 등)을 정하고, 1~2개 베이스라인을 적합·진단한다. | 04, 05_model_grid, 재현성 명세 | 05, 06_evaluation_matrix(초안) |
| **3** | 모델을 비교(ΔAIC, LRT 등)하고, 로버스트니스·안정성을 검토해 최종 모델을 결정한다. | 05, 06 | 06(비교), 07_decision_log |
| **4** | (선택) 정규화, 상호작용, 시간 변수 등 고급 사양을 필요 시 추가한다. | 05, 06, 07 | 확장 모델·과적합 통제 문서 |
| **5** | 결과를 한데 모아 보고서로 정리하고, 한계·재현성·다음 단계를 명시한다. | 01~07, 설계 템플릿 | 08_results_report |

---

## 왜 이 순서인가 (납득 구조)

- **계약(0) 먼저:** “무엇을 측정한 데이터인지”가 정해져야 이후 기술통계·모델 해석이 의미 있다.  
- **특성·가설(1) 다음:** 데이터를 본 뒤 가설을 세우고, 가설에 맞는 모델을 고른다(데이터 기반 가설 → 모델 선택).  
- **모델·평가·결정(2–3):** 선택한 모델을 적합하고, 비교·로버스트니스로 “어떤 결론을 내릴지”를 증거와 불확실성(CI)으로 정리한다.  
- **보고(5):** 위 과정과 증거·한계를 고정 형식(08 템플릿)으로 남겨 재현과 검토를 가능하게 한다.

---

## 규칙과의 대응 (요약)

- **R0 Phase lock:** 위 순서를 지키고, 해당 단계 산출물이 채워지기 전에 다음 단계로 넘어가지 않는다.  
- **R1 Evidence:** 데이터에 대한 주장은 스키마·SQL·집계 등 증거를 반드시 인용한다.  
- **R3 Uncertainty:** 추정은 점추정 + 95% CI(또는 SE)를 함께 보고한다.  
- **R6 Reproducibility:** 사용 라이브러리·함수·파라미터는 design/00_reproducibility_spec에만 두고, 랜덤 시드를 고정한다.
